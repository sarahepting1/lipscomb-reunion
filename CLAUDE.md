# Lipscomb Family Reunion Website

## What this project is

A family reunion website replacing the static site at lipscombfamilyreunion.com. It serves ~1,070 family members (living and deceased) from a merged genealogy dataset. Family members can browse a directory, view family relationships, and submit life events (births, deaths, marriages, address changes) through forms. Submissions are held for admin approval before touching live data.

The site owner (Sarah) is a Salesforce consultant — comfortable with data models, flows, and admin concepts. Explain web-dev-specific concepts (RLS, environment variables, deployments) briefly when they first come up, but don't over-explain data concepts.

## Stack

- **Next.js** (App Router, TypeScript) — hosted on **Vercel** free tier
- **Supabase** free tier — Postgres database + Auth
- **Tailwind CSS** for styling
- No other services. Keep dependencies minimal.

See `AGENTS.md` for Next.js-version-specific agent rules generated by the scaffold (breaking-change warnings, doc pointers).

## Source data

`lipscomb_family_data.json` in the repo root. Structure:

- `people[]` — 1,070 records with stable IDs (`P0001`–`P1070`). Each has: `name`, `aka`, `sex`, `birth {date, date_raw, year, qualifier, location}`, `death` (same shape, or null), `address {street, city, state, zip}` (or null), `status`, and denormalized `parents[]`, `children[]`, `spouses[]` arrays.
- `marriages[]` — 366 records with `husband_id`/`wife_id` (nullable when ambiguous), dates, ending status.
- `data_quality` — known issues (ambiguous links, duplicate names). Do not try to fix these; import as-is.

**IDs are the source of truth, never names** — 17 names are shared by multiple distinct people.

`status` values: `deceased` (recorded death), `presumed_deceased` (born 100+ years ago), `presumed_living`, `unknown`.

## Privacy rules (non-negotiable)

- **Public pages** (no login): home/reunion info, recipes, and directory entries ONLY for people with status `deceased` or `presumed_deceased`. Never show addresses on public pages, even for the deceased.
- **Family-gated pages** (login required): full directory including living people, addresses, full birthdates, and all forms.
- Auth model: the original shared family account (email + password Sarah distributes) still works, but self-service registration was added post-v1 (see Phase 6). New signups require a `FAMILY_SIGNUP_CODE` (separate from the shared account's password) to keep strangers out.
- Enforce gating server-side (RLS policies + server components), not just by hiding links.

## Database schema

```sql
people (
  id text primary key,            -- 'P0001' format; new people get 'P1071'+
  name text not null,
  aka text,
  sex text,
  birth_date date,                -- null when only year/partial known
  birth_date_raw text,            -- always preserve original string
  birth_year int,
  birth_location text,
  death_date date,
  death_date_raw text,
  death_year int,
  death_location text,
  street text, city text, state text, zip text,
  status text not null check (status in ('deceased','presumed_deceased','presumed_living','unknown')),
  created_at timestamptz default now(),
  updated_at timestamptz default now()
)

marriages (
  id uuid primary key default gen_random_uuid(),
  husband_id text references people(id),
  husband_name text not null,     -- keep name even when id unresolved
  wife_id text references people(id),
  wife_name text,
  marriage_date date,
  marriage_date_raw text,
  ending_status text
)

parent_child (
  id uuid primary key default gen_random_uuid(),
  parent_id text references people(id),
  parent_name text not null,
  child_id text references people(id),
  child_name text not null,
  relationship text               -- 'Natural', 'Adopted', etc.
)

submissions (
  id uuid primary key default gen_random_uuid(),
  event_type text not null check (event_type in ('birth','death','marriage','address_update','correction')),
  person_id text references people(id),   -- null for new people (births/marriages)
  payload jsonb not null,                 -- the submitted fields
  submitter_name text not null,
  submitter_contact text,                 -- email or phone, free text
  notes text,
  status text not null default 'pending' check (status in ('pending','approved','rejected')),
  reviewed_at timestamptz,
  created_at timestamptz default now()
)

user_people (                             -- added in Phase 6 (self-service registration)
  user_id uuid primary key references auth.users(id) on delete cascade,
  person_id text references people(id),   -- which directory record this account belongs to
  created_at timestamptz default now()
)
```

RLS: anonymous role can SELECT people where status in ('deceased','presumed_deceased') — excluding address columns (use a public view). Authenticated role can SELECT everything and INSERT into submissions. Only the service role (admin actions via server routes) can UPDATE people or approve submissions. Authenticated users can SELECT/INSERT/UPDATE only their own row in `user_people` (`auth.uid() = user_id`).

## Pages

1. `/` — public. Reunion info (date/location placeholder Sarah will edit), welcome text, link to recipes and directory.
2. `/recipes` — public. Placeholder for now; recipe content arrives in a later phase as markdown.
3. `/directory` — gated. Searchable, alphabetical list of all people. Search by name (case-insensitive, matches aka too). Show name, birth–death years, city/state.
4. `/person/[id]` — gated. Full profile: dates, locations, address, parents, spouses, children, and a Family Tree section (ancestors/descendants, 3 generations out) — each relationship linked to its person page when an ID exists, plain text when not.
5. `/submit` — gated. One form, event-type picker changes the fields:
   - **Birth**: child name, birthdate, parents (searchable person picker + free text fallback), location
   - **Death**: person (picker), date, location
   - **Marriage**: two people (picker + free text), date
   - **Address update**: person (picker), new address fields
   - **Correction**: person (picker), free-text description
   - All types: submitter name (required), contact (optional), notes
6. `/admin` — gated by a separate admin password (env var `ADMIN_PASSWORD`, checked server-side; don't build a second auth system). Lists pending submissions with the parsed payload; Approve applies the change to the data tables inside a transaction, Reject just marks it. Show approved/rejected history below.
7. `/login` — the shared family login (still works).
8. `/signup` — self-service registration. Requires email, password, and the `FAMILY_SIGNUP_CODE` (a short code Sarah distributes, separate from the shared account's password). On success, redirects to `/link-person` (or shows a "check your email" message if Supabase's email confirmation is required first).
9. `/link-person` — gated. Search-and-pick (no free-text fallback — must match a real directory record) to link the logged-in account to their own `people` row via `user_people`. Skippable; reachable anytime later via the "Link My Profile" nav link.

## Build phases — do these in order, one at a time

Confirm each phase works before starting the next. Sarah will test between phases.

- **Phase 1**: Scaffold Next.js app, Supabase client setup, schema migration SQL, seed script that loads the JSON (`npm run seed`). Definition of done: all 1,070 people + marriages + parent links queryable in Supabase table editor, counts match the JSON's `counts` object.
- **Phase 2**: Login + directory + person pages with privacy gating. Done: logged-out user sees only deceased people and no addresses anywhere; logged-in sees everything; relationship links navigate correctly.
- **Phase 3**: Submission form. Done: each event type writes a well-formed row to `submissions`.
- **Phase 4**: Admin review page. Done: approving a death submission sets the person's death fields and flips status to `deceased`; approving a birth creates a new person (next sequential ID) plus parent_child rows; approving an address update overwrites address fields; everything else applies sanely; rejected submissions change nothing.
- **Phase 5**: Deploy to Vercel, env vars set, custom domain instructions for lipscombfamilyreunion.com.
- **Phase 6** (post-v1 addition): self-service registration (`/signup`, `FAMILY_SIGNUP_CODE` gate) with a required-or-skippable person-linking step (`/link-person`, `user_people` table), a per-person Family Tree view (ancestors/descendants, 3 generations), and a search fix so multi-word queries match non-contiguous name parts (e.g. "Sarah Epting" now matches "Sarah Hope Epting").

## Conventions

- Commit at the end of every phase with a clear message.
- Keep secrets in `.env.local` (gitignored): `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `ADMIN_PASSWORD`, `FAMILY_SIGNUP_CODE`. Never commit keys; never expose the service role key to the client.
- On Windows, never pipe secrets into `vercel env add` via PowerShell's pipe (`Write-Output $val | vercel env add ...`) — it can prepend a UTF-8 BOM that silently corrupts the value. Write the value to a file with `[System.IO.File]::WriteAllText($path, $value, [System.Text.UTF8Encoding]::new($false))` and feed it in via `cmd /c "vercel env add NAME env < file"` instead. After changing any `NEXT_PUBLIC_` var, redeploy with `--force` since Next.js inlines those at build time and a cached build won't pick up the change.
- Server components by default; client components only where interactivity requires.
- Design: warm, family-friendly, readable at large font sizes (many users are older). Works well on phones — most family members will visit on mobile. No heavy frameworks or animations.
- When something in this brief conflicts with what Sarah asks for in session, ask her — her word wins.
